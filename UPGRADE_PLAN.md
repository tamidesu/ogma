# OGMA — Comprehensive Upgrade Plan
**From scripted quiz → open-world profession simulator**

> Decisions locked: **MVP = Doctor profession only**, **Curated tagged image library**, **Free-text input + AI-suggested action chips**, **Images will be hand-generated by the user from descriptions in this document**.

---

## 0. Executive Summary

Today OGMA is a polished but fundamentally **deterministic quiz**: JSON fixtures define `Step → DecisionOption[]`, the engine picks the next step from a hardcoded `next_step_key`, and AI is invoked only after the choice to write mentor commentary. Characters are static cards with one of four colored states.

To deliver the pitch ("you freely write what you want to do, several AI agents react, the world changes under your decisions, grounded in real laws and protocols") we replace the rigid step machine with a **multi-agent orchestrator**, introduce a **stateful unified NPC** with persistent emotion + memory, drive visuals from a **tagged curated image library**, and rebuild the simulation page around a **free-text input + cinematic image stage + live NPC panel**.

The MVP target is a single profession — **Doctor** — built end-to-end at production quality. Once the pattern is proven, porting to lawyer / engineer / business_manager is a content + asset effort, not a re-architecture.

## 2. Target Vision

### 2.1 The session loop (per turn)

```
┌──────────────────────────────────────────────────────────────────────────┐
│  USER reads the case file / sees the scene image / sees NPC's last line  │
│              ↓                                                           │
│  USER types a free-text action (or picks one of 4 suggested chips)       │
│              ↓                                                           │
│  ─────────── Agent Pipeline (parallel where possible) ───────────────    │
│                                                                          │
│  1. Action Interpreter   → structured intent {verb, target, params,      │
│                            plausibility, requires_clarification}         │
│  2. Domain Validator     → RAG retrieves relevant protocols/laws/        │
│      (RAG-grounded)        guidelines; LLM judges legality, correctness, │
│                            standard-of-care alignment + citations         │
│  3. NPC Director         → updates NPC emotion state; generates          │
│                            in-character utterance + body-language hint   │
│  4. Consequence Engine   → applies world-state deltas, advances          │
│                            world-clock, may spawn a new complication     │
│  5. Visual Director      → picks an image from the library matching     │
│                            new world state + NPC emotion + scene         │
│                                                                          │
│  ↓ STREAMED TO CLIENT VIA SSE in this order ↓                            │
│                                                                          │
│  6. Mentor Coach         → async background, streams in last (existing  │
│                            FeedbackGenerator, lightly adapted)           │
└──────────────────────────────────────────────────────────────────────────┘
```

Every part of this is data-driven. **No more hardcoded `next_step_key`.** The "next step" is whatever world state the Consequence Engine produces.

### 2.2 Termination

A scenario brief defines `success_criteria` and `failure_criteria` as predicate expressions over the world state (e.g. `patient.alive == true && patient.diagnosis_correct == true && time_elapsed_min < 90`). After each turn, the orchestrator evaluates them. If matched → terminal turn → final mentor report → results page. If neither matched after `max_turns` (e.g. 12), the brief defines a `timeout` resolution.

### 2.3 What "open world" actually means here

It does **not** mean "the LLM hallucinates anything." It means:
- The world state is a **structured, typed object** the agents read and write.
- The user can type any plausible action, but the Validator filters: "you can't 'order an MRI' if the radiology department is closed and no on-call radiologist is reachable."
- The library of possible world states is bounded by the scenario brief (NPC roster, available resources, possible complications).
- Within those bounds, the agents have full freedom to compose.

This is the right tradeoff: real enough to feel open, structured enough to evaluate and to render with curated images.

## 3. Architecture: The Agent Pipeline

### 3.1 New module: `app/agents/`

```
├── schemas.py                   # Shared dataclasses (Intent, NPCUpdate, …)
└── prompts/
    ├── interpreter.system.md

    ├── consequence_engine.system.md
    └── mentor_coach.system.md   # adapted from existing prompt_builder.py
```

### 3.2 Shared schema (`app/agents/schemas.py`)

```python
@dataclass(frozen=True)
class UserAction:
    raw_text: str
    locale: str

@dataclass(frozen=True)
class Intent:
    verb: str                       # "order", "ask", "examine", "console", ...
    target: str                     # "patient.airway", "blood_bank", "family", ...
    parameters: dict[str, Any]      # {"medication": "epinephrine", "dose_mg": 1}
    plausibility: float             # 0..1 — Validator may downgrade later
    requires_clarification: str | None

@dataclass(frozen=True)
class ValidationResult:
    is_legal: bool
    is_standard_of_care: bool
    severity_if_wrong: Literal["minor","moderate","severe","critical"]
    citations: list[Citation]       # {source, snippet, url_or_ref}
    coach_note: str                 # one sentence, rendered later

@dataclass(frozen=True)
class NPCUpdate:
    emotion_delta: dict[str, int]   # {"trust": -10, "fear": +20}
    new_state_label: str            # "anxious_distressed"
    utterance: str                  # what the NPC says, in scenario locale
    body_language: str              # one short phrase for visual director hint

@dataclass(frozen=True)
class WorldDelta:
    metric_delta: dict[str, float]  # patient_stability, diagnosis_accuracy, ...
    state_set: dict[str, Any]       # arbitrary world flags
    time_advance_min: int
    new_complication: dict | None   # spawned event, e.g. "blood_bank.units = 2"
    skill_xp: list[SkillXPGain]

@dataclass(frozen=True)
class TurnResult:
    intent: Intent
    validation: ValidationResult
    npc_update: NPCUpdate
    world_delta: WorldDelta
    image_id: str
    is_terminal: bool
    termination_reason: str | None
```

### 3.3 Orchestrator (sequencing rules)

- **Sequential, not free-for-all.** Order matters because the Validator's verdict feeds into NPC mood (a malpractice attempt makes the patient panic), and the NPC's new emotion feeds image selection.
- **Streaming via SSE.** Each agent's output is pushed to the client as soon as it's ready. The user sees: intent recognized → validator verdict → NPC reacts → world updates → image transitions → mentor coach types in.
- **Single LLM provider for all agents on MVP** — Groq `llama-3.3-70b-versatile`. JSON mode for every call. Temperature ~0.3 for Interpreter/Validator/Consequence (deterministic-ish), ~0.7 for NPC Director and Mentor Coach (creative).
- **Concurrency-safe.** Agents read an immutable snapshot of session state; the orchestrator merges deltas at the end of the turn under a Redis lock keyed by `session_id`.
- **Failure handling.** If an agent times out or returns invalid JSON twice (existing tenacity retry pattern), the orchestrator returns a safe fallback (`Intent` becomes `verb=clarify`, NPC stays neutral, world unchanged, image = previous frame). User is shown a "system needed a moment, please rephrase" toast.

### 3.4 Agent prompts — the bones

**Action Interpreter** — system prompt establishes: "You parse a doctor trainee's free-text action into a structured intent. You never refuse — even nonsense gets a low plausibility score. Output JSON only."

**Domain Validator** — system prompt: "You are a clinical reviewer. Given the trainee's intent and the retrieved guideline excerpts, judge: is this legal under medical practice law? Is this standard-of-care for this presentation? What's the severity if wrong? Cite the specific protocol/law section. Output JSON only."

**NPC Director** — receives: NPC personality JSON, NPC current emotion, NPC memory (last N turns), validator verdict, intent. Outputs emotion deltas + a single line of dialogue + body language tag. The NPC is a **person**, not a quest-giver: agrees, disagrees, gets scared, gets angry, asks for clarification, references something the user said two turns ago.

**Consequence Engine** — receives: validator verdict, NPC update, current world state, scenario brief constraints. Outputs metric deltas, state flags, optional new complication, time advance. This is the "world reacts" agent. It is constrained: it can only modify metrics defined in the brief, only set flags in the brief's state schema, only spawn complications from the brief's `complication_pool`. Keeps things grounded.

**Visual Director** — pure retrieval, no LLM call by default (saves latency + cost). Takes the new scene category + NPC emotion + active world flags → builds a query string → BM25 over the image manifest → picks top-1. If no good match (score below threshold), falls back to a profession-default scene image. Optional: cache the (state_hash → image_id) mapping in Redis for repeat hits.

### 3.5 Why this architecture is right

- **Grounded** — Validator + RAG + bounded Consequence Engine prevent the "AI hallucinates a magic medication" failure mode.
- **Reactive** — NPC has memory and emotion, so the same situation feels different turn-to-turn.
- **Cinematic** — Visual Director gives the user a constantly-shifting backdrop without per-turn image generation cost.
- **Streamable** — Agents finish at different times; SSE lets the UI feel alive ("intent recognized" appears in 0.5s while the deeper analysis is still cooking).
- **Replaceable** — Each agent is a pure function of state + context. We can swap models, prompts, even strategies (e.g. NPC Director could later become a fine-tuned 7B model) without touching anything else.

### 4.2 Manifest schema

```json
{
  "images": [
    {
      "id": "doctor.er_trauma_bay.chaos.dawn",
      "file": "doctor/er_trauma_bay_chaos_dawn.jpg",
      "profession": "doctor",
      "scene_category": "er_trauma_bay",
      "mood": "panic",
      "characters_present": ["unconscious_patient", "nurse", "trainee_doctor_pov"],
      "world_flags_match": ["mass_casualty_active", "additional_staff_called=false"],
      "tags": ["dawn", "fluorescent", "blood", "stretcher", "monitors", "wide-shot"],
      "description": "Searchable text — see prompt list in §11."
    },
    ...
  ]
}
```

### 4.3 Selection algorithm

1. Build a query string from `scene_category + " " + mood + " " + active_flags + " " + npc_emotion + " " + body_language`.
2. BM25 over `tags + description + characters_present` of all images for the current profession.
3. Apply a hard filter: `scene_category` must match (no exam-room image during a code-blue).
4. If top candidate scores below `min_score` (say 2.0), fall back to a profession-level "neutral" image keyed `doctor.transition.neutral`.
5. Return `{id, url, alt_text}` — alt_text from the description, used for accessibility.

No LLM call needed for selection on MVP. We can swap to a small embedding model later if BM25 proves too brittle.

### 4.4 Frontend rendering

A single `<SceneStage />` component holds the current image. Transitions:
- Crossfade 600ms when image changes.
- Subtle Ken-Burns zoom (1.0 → 1.04 over 8s) so static images feel alive.
- Optional desaturation overlay when world state turns dire (e.g. patient deteriorating).

## 5. NPC System: Unified Stateful Character

### 5.1 NPC definition (per scenario brief)

```python
@dataclass
class NPCDefinition:
    id: str                          # "elena_lymphoma"
    display_name: str                # "Elena, 45"
    role: str                        # "Patient (suspected lymphoma)"
    backstory: str                   # multi-paragraph; system-prompted into NPC Director
    personality_traits: dict[str,int] # Big-Five-ish, 0..100
    initial_emotion: dict[str,int]   # {trust:60, fear:30, anger:10, hope:50}
    voice_directives: str            # "Speaks softly, asks lots of clarifying questions, occasionally interrupts."
    memory_seeds: list[str]          # things the NPC "knows" at turn 0
    avatar_image_id: str             # references a curated avatar image
```

### 5.2 NPC state per session (Redis-cached, Postgres-persisted)

```python
@dataclass
class NPCState:
    definition_id: str
    emotion: dict[str,int]              # decays toward baseline each turn (10%)
    relationship_score: int             # -100..+100 — long-arc trust with the user
    memory: list[NPCMemoryItem]         # rolling window of last ~12 turns
    current_label: str                  # "anxious_distressed" — drives visuals
```

`NPCMemoryItem`: `{turn_index, user_intent_summary, npc_response_summary, emotion_at_time}`. Capped at 12; older items summarized into a single "long-term memory" string by a periodic compaction job (or just trimmed for MVP).

### 5.3 Frontend `<NPCPanel />`

Replaces the current static `CharacterCard` row. Contents:
- Avatar (image from library; can change between e.g. "calm_eye_contact" and "distressed_looking_away")
- Name + role
- Emotion bar — four mini-bars (trust / fear / anger / hope), animated on change
- Last utterance — speech bubble, fade-in, lingers until next turn
- Body-language footnote — italic, small ("She glances toward the door")
- Memory toggle — expands a scroll of past exchanges (read-only)

## 7. Database Schema Changes

New Alembic migration: **`0003_open_world_simulation.py`**.

### 7.1 New tables

```sql
-- Scenario briefs replace the rigid Step-table for the new flow
CREATE TABLE scenario_briefs (
    id              UUID PRIMARY KEY,
    profession_id   UUID REFERENCES professions(id),
    slug            TEXT UNIQUE NOT NULL,
    title           TEXT NOT NULL,
    case_file_md    TEXT NOT NULL,           -- the "case file" the user reads first
    initial_world_state JSONB NOT NULL,      -- typed schema below
    success_criteria_jsonlogic JSONB,        -- predicate over world_state
    failure_criteria_jsonlogic JSONB,
    timeout_resolution TEXT,                 -- narrative for max_turns hit
    max_turns       INT DEFAULT 12,
    npc_definition  JSONB NOT NULL,          -- NPCDefinition serialized
    complication_pool JSONB NOT NULL,        -- list of possible spawns
    knowledge_tags  TEXT[],                  -- which RAG tags are in scope
    locale          TEXT DEFAULT 'kk',
    version         INT DEFAULT 1,
    created_at      TIMESTAMPTZ
);

-- One row per turn
CREATE TABLE agent_turns (
    id                 UUID PRIMARY KEY,
    session_id         UUID REFERENCES sim_sessions(id) ON DELETE CASCADE,
    turn_index         INT NOT NULL,
    user_input         TEXT NOT NULL,
    intent_json        JSONB NOT NULL,
    validation_json    JSONB NOT NULL,
    npc_update_json    JSONB NOT NULL,
    world_delta_json   JSONB NOT NULL,
    image_id           TEXT,
    mentor_feedback_id UUID REFERENCES ai_feedback(id),
    is_terminal        BOOL DEFAULT FALSE,
    termination_reason TEXT,
    created_at         TIMESTAMPTZ,
    UNIQUE(session_id, turn_index)
);

-- Append-only world state snapshots for replay/debugging
CREATE TABLE world_state_snapshots (
    id          UUID PRIMARY KEY,
    session_id  UUID REFERENCES sim_sessions(id) ON DELETE CASCADE,
    turn_index  INT NOT NULL,
    state_json  JSONB NOT NULL,
    created_at  TIMESTAMPTZ
);

-- NPC memory persistence
CREATE TABLE npc_states (
    session_id        UUID PRIMARY KEY REFERENCES sim_sessions(id) ON DELETE CASCADE,
    definition_id     TEXT NOT NULL,
    emotion_json      JSONB NOT NULL,
    relationship_score INT NOT NULL DEFAULT 0,
    memory_json       JSONB NOT NULL,
    current_label     TEXT,
    updated_at        TIMESTAMPTZ
);

-- Image manifest mirrored in DB so we can search/edit without redeploys
CREATE TABLE image_assets (
    id                 TEXT PRIMARY KEY,        -- "doctor.er_trauma_bay.chaos.dawn"
    file_path          TEXT NOT NULL,
    profession_slug    TEXT NOT NULL,
    scene_category     TEXT NOT NULL,
    mood               TEXT NOT NULL,
    characters_present TEXT[],
    world_flags_match  TEXT[],
    tags               TEXT[],
    description        TEXT NOT NULL,
    bm25_searchable    TSVECTOR GENERATED ALWAYS AS (...) STORED,
    created_at         TIMESTAMPTZ
);
CREATE INDEX idx_image_assets_search ON image_assets USING GIN(bm25_searchable);
CREATE INDEX idx_image_assets_profession_scene ON image_assets(profession_slug, scene_category);
```

- `decision_log`: keep for legacy. New flow writes `agent_turns` instead.
- `ai_feedback`: unchanged; mentor coach still writes here.

### 7.3 Backward compatibility

Old `/decide` endpoint + scripted scenarios stay working. Existing seed data (`tests/fixtures/scenarios/*.json`) becomes "guided mode" content for users who want a tutorial-style intro before the open world.

## 13. Success Criteria for the Demo

The demo is a winner if a stranger sitting down can:

1. Pick "Doctor — The Uncertain Diagnosis" from the briefs list.
2. Read a 2-paragraph case file that feels like a real chart note.
3. Type any plausible action ("ask Elena about her family history of cancer") and within 4 seconds see:
   - Their intent acknowledged
   - The validator confirm it's a reasonable history-taking step (with a citation)
   - Elena respond in character ("My mother had breast cancer at 50…") with her trust bar nudging up
   - The scene image shift to a quieter, more intimate framing
4. Make 3–5 turns of choices, each visibly changing the world.
5. Receive a final mentor report that references their actual decisions, not generic.
6. Be unable to "break" the simulation by typing nonsense — the validator catches it.

## 14. Risks & Mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Latency budget blown — agent pipeline >6s per turn | High | Stream incrementally; run Interpreter and Validator in parallel after the first; small prompts; cache suggested actions; precompute Visual Director (BM25 is microseconds). |
| LLM outputs invalid JSON | Medium | Existing `groq_adapter` retry pattern + tenacity; one re-prompt with "your previous output was invalid, here is the schema again"; final fallback to safe `Intent(verb="clarify")`. |
| NPC drifts out of character over many turns | Medium | NPC memory window + explicit personality re-injection in every `npc_director` call; periodic "compaction" turn that summarizes long memory. |
| Validator becomes a wet blanket — rejects creative actions | Medium | Split severity: only `severity in (severe, critical)` actually blocks; minor/moderate proceeds with a coach note. |
| Image library too sparse → fallback used too often | Medium | §11.12 priority order ensures the most common scenes are filled first; instrument fallback rate per scenario; expand library where it spikes. |
| Cost runaway from 5 LLM calls per turn | Medium | Groq is cheap relative to OpenAI; cap per-call max_tokens (Interpreter 200, Validator 400, NPC 250, Consequence 350, Mentor 600); `suggested-actions` cached aggressively. |
| Free-text users feel lost at turn 1 | Low (mitigated by chips) | Always show 4 suggested actions; first turn's chips authored manually in the brief, not LLM-generated, to guarantee quality. |
| Backward compatibility with existing scripted scenarios | Low | Legacy `/decide` endpoint and `OptionCard` UI kept behind `mode='guided'`; new flow is `mode='open_world'`. |
